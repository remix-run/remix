Added `Database.wipe()`, `Database.migrate()`, `Database.migrationStatus()`, and `Database.reset()` along with the typed `runRemixDb()` API from `remix/data-table/cli`. The migrate `to` target accepts a bare migration id (`20260301113000`) or the full `id_name` directory form (`20260301113000_add_user_status`). Migration status reports applied journal entries missing from the current migration set; `up()` runs reject missing applied migrations before executing SQL, while `down()` skips orphaned journal entries so migrations that are still present can be rolled back. Apps can export `db`, `getMigrations`, and an optional `seed` function from `app/db.ts` for use by `remix db` (see #11608).

BREAKING CHANGE: The `acquireMigrationLock()`/`releaseMigrationLock()` adapter hooks were replaced by the scoped `withMigrationLock()` contract, which runs the migration and journal lifecycle on the connection that owns the lock and must release the lock whether the callback resolves or rejects. Adapter authors must implement `withMigrationLock()`; adapters that still only implement the old hooks will run migrations without locking.
