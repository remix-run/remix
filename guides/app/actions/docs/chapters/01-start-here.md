## What is Remix?

Remix is a TypeScript framework built around the web's request and response model. A server receives a Web `Request`, Remix matches it to a typed route, a controller handles the request, and the app returns a Web `Response`.

The `remix` package gives you the full stack: a server adapter, fetch router, typed routes, middleware, controllers, response helpers, and testing utilities. On the UI side, it includes server-rendered components, frames for partial UI, browser-loaded client entries, client navigation, CSS-first animation helpers, source-served assets, and first-party primitives such as menus, popovers, listboxes, selects, and tabs.

Later chapters use the same package for form parsing and validation, cookies and sessions, credentials auth, OAuth/OIDC, CSRF/CORS/COP protection, file uploads, and database queries, transactions, and migrations.

Each piece of `remix` is also designed to stand on its own, so you can use the full framework or adopt smaller pieces when an app only needs routing, UI, API endpoints, auth, database access, or browser behavior.

The design of Remix comes from six core principles:

1. **Agent-First Development.** Remix optimizes source code, documentation, tooling, and abstractions for LLMs, and provides primitives for using models inside the products you build.
2. **Build on Web APIs.** Remix builds on `Request`, `Response`, `URL`, `FormData`, headers, cookies, and JavaScript because shared platform APIs reduce context switching across the full stack while keeping your application portable.
3. **Religiously Runtime.** Remix APIs do not depend on bundlers, type generation, or static analysis. Routes, middleware, controllers, and tests should run as ordinary runtime code.
4. **Avoid Dependencies.** Remix chooses dependencies carefully, wraps them behind its own boundaries, and works toward replacing them with focused packages when the framework needs long-term control.
5. **Demand Composition.** Remix packages are single-purpose, replaceable, and useful on their own. You can add or remove pieces as an app changes, while tightly coupled modules belong together.
6. **Distribute Cohesively.** Remix keeps the ecosystem easy to learn and use by distributing the composable pieces through one `remix` package and one documentation path.

The fastest way to understand Remix is to build the request path once: create a server, define routes, connect a router, and return a response.

## Quickstart: create and run a Remix app {#quickstart}

The quickest way to start a Remix app is with the `remix` CLI. It creates a small full-stack app with a Node server, typed routes, render middleware, a server-rendered home page, source-served assets, and one hydrated component.

```sh
npx remix@next new my-remix-app
```

Install the project dependencies:

```sh
cd my-remix-app
npm i
```

Open `package.json` and you will see one runtime dependency: `remix`. Your app uses that package for the server adapter, router, middleware, UI, assets, and client entry.

Start the dev server:

```sh
npm run dev
```

The server prints the local URL when it starts. By default, the dev server listens on [http://localhost:44100](http://localhost:44100).

![Default Remix home page](/images/app/actions/docs/chapters/01-start-here/remix-template.png)

Your new app includes an Agent Skill and a home page with prompts you can copy into an agentic coding environment. Try one if you want to explore on your own. When you are ready, come back here and we'll take a tour of the files.

## Project tour: where code lives {#project-tour}

A typical Remix application keeps the request flow visible in the file structure. An HTTP request enters `server.ts`, is handed to `app/router.ts`, runs through middleware, is matched against the route definitions in `app/routes.ts`, is handled by a controller action in `app/actions/`, and returns a Web `Response`. Often, that response is HTML generated by a Remix component, but the same path can return redirects, JSON, files, streams, errors, or any other Web response.

That flow gives all of your code a clear place to live. Keep route definitions and URL helpers in routes, connect middleware, routes, and controllers in the router, run cross-cutting request logic in middleware, put route-specific behavior in actions, share views and layouts through UI, and put browser behavior in assets.

```txt
my-remix-app/
├── server.ts         # request entrypoint for this runtime
└── app/
    ├── routes.ts     # route definitions, route map, and URL helpers
    ├── router.ts     # middleware, routes, and controller mapping
    ├── middleware/   # request middleware and context providers
    ├── actions/      # controllers and route actions
    ├── ui/           # shared views, layouts, and Remix components
    ├── assets.ts     # browser module server configuration
    └── assets/       # browser entrypoints and client behavior
```

### `server.ts` hands requests to Remix

`server.ts` is the request entrypoint for the runtime you choose. In Node, it listens for HTTP requests and passes each one to `router.fetch(request)`. Keep runtime concerns here: ports, shutdown, and last-chance error logging. Route behavior belongs in controllers.

The [Server Runtime](/docs/server-runtime#the-node-server-entry) chapter digs into runtime adapters and [`createRequestListener`](/docs/server-runtime#createrequestlistener).

### `app/routes.ts` defines the route map

Routes are ordinary TypeScript values. A route map names URL patterns and creates href helpers for links, redirects, tests, and frame sources.

The [Core App Structure](/docs/core-app-structure#routes-as-the-url-contract) chapter covers route maps and [route builders](/docs/core-app-structure#route-builders-route-get-post-put-del-form-resources) in more detail.

### `app/router.ts` connects middleware, routes, and controllers

The router installs middleware and maps route leaves to controller actions. You will touch this file when adding app-wide middleware or mapping a new route group.

The [Server Runtime](/docs/server-runtime#middleware-ordering) chapter covers middleware order, and [Core App Structure](/docs/core-app-structure#controllers-and-actions) covers controller mapping.

### `app/middleware/` runs cross-cutting request logic

Middleware runs before and/or after controller actions. Use it for cross-cutting request behavior such as static files, sessions, auth, form parsing, rendering, logging, compression, and security checks.

The [Middleware ordering](/docs/server-runtime#middleware-ordering) and [Typed request context](/docs/server-runtime#typed-request-context) sections go deeper.

### `app/actions/` owns route behavior

`app/actions/` is route-owned server code. Despite the name, it is not only for form submissions; it is where controllers, route actions, and route-local helpers live. A controller action reads request data and returns a `Response`.

The next section maps a request to an action. The [Controllers and actions](/docs/core-app-structure#controllers-and-actions) and [Responses, redirects, headers, and errors](/docs/core-app-structure#responses-redirects-headers-and-errors) sections cover the larger model.

### `app/ui/` holds shared views and layouts

`app/ui/` is for UI shared across routes, such as the document shell, app layout, and reusable page pieces. Remix components render server HTML; keep route-specific UI near the route until more than one route needs it.

The [Rendering UI](/docs/rendering-ui#the-remix-component-model) chapter covers the component model, document shells, styling, and first-party UI components. We'll also build a small page in [Build your first page](#build-your-first-page).

### `app/assets.ts` and `app/assets/` serve browser modules

In this guide, “assets” means browser-facing code and files. `public/` is for files served as-is, such as favicons, images, and plain CSS. `app/assets.ts` configures Remix's source asset server, and `app/assets/` contains TypeScript or JSX modules that Remix should load in the browser.

The [Files and Assets](/docs/files-and-assets#static-files-vs-source-served-assets) chapter explains static files, source-served assets, file maps, and production options; [Interactivity](/docs/interactivity#cliententry) covers client entries.

## Map your first request handler {#map-your-first-request-handler}

A Remix request handler is a route definition plus a controller action. The route says which URL and method you accept; the action reads the request context and returns a response.

Start by adding an album route with a URL param:

```ts filename=app/routes.ts lines=[6]
import { get, route } from 'remix/routes'

export const routes = route({
  assets: get('/assets/*path'),
  home: '/',
  album: get('/albums/:albumId'),
})
```

`album` is the route name. `get('/albums/:albumId')` means Remix should match `GET` requests like `/albums/thriller`, and `:albumId` becomes `params.albumId` in the action. The route name also gives you an href helper: `routes.album.href({ albumId: 'thriller' })`.

Add the matching action in `app/actions/controller.tsx`:

```tsx filename=app/actions/controller.tsx lines=[17-19]
import { createController } from 'remix/router'

import { assetServer } from '../assets.ts'
import { routes } from '../routes.ts'
import { HomePage } from '../ui/scaffold-home-page.tsx'

export default createController(routes, {
  actions: {
    async assets(context) {
      return (
        (await assetServer.fetch(context.request)) ?? new Response('Not Found', { status: 404 })
      )
    },
    home(context) {
      return context.render(<HomePage />)
    },
    album({ params }) {
      return new Response(`Album: ${params.albumId}`)
    },
  },
})
```

The action returns a plain Web `Response`. There is no page component yet; this is the server path first. Restart the dev server if it is not already watching, then open [http://localhost:44100/albums/thriller](http://localhost:44100/albums/thriller) in your browser.

You should see the text response on the page:

```txt
Album: thriller
```
