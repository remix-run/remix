## What is Remix?

Remix is a TypeScript framework built around the web's request and response model. A server receives a Web `Request`, Remix matches it to a typed route, a controller handles the request, and the app returns a Web `Response`.

The `remix` package gives you the full stack: a server adapter, fetch router, typed routes, middleware, controllers, response helpers, and testing utilities. On the UI side, it includes server-rendered components, frames for partial UI, browser-loaded client entries, client navigation, CSS-first animation helpers, source-served assets, and first-party primitives such as menus, popovers, listboxes, selects, and tabs.

Later chapters use the same package for form parsing and validation, cookies and sessions, credentials auth, OAuth/OIDC, CSRF/CORS/COP protection, file uploads, and database queries, transactions, and migrations.

Each piece of `remix` is also designed to stand on its own, so you can use the full framework or adopt smaller pieces when an app only needs routing, UI, API endpoints, auth, database access, or browser behavior.

The design of Remix comes from six core principles:

1. **Agent-First Development.** Remix optimizes source code, documentation, tooling, and abstractions for LLMs, and provides primitives for using models inside the products you build.
2. **Build on Web APIs.** Remix builds on `Request`, `Response`, `URL`, `FormData`, headers, cookies, and JavaScript because shared platform APIs reduce context switching across the full stack while keeping your application portable.
3. **Religiously Runtime.** Remix APIs do not depend on bundlers, type generation, or static analysis. Routes, middleware, controllers, and tests should run as ordinary runtime code.
4. **Avoid Dependencies.** Remix chooses dependencies carefully, wraps them behind its own boundaries, and works toward replacing them with focused packages when the framework needs long-term control.
5. **Demand Composition.** Remix packages are single-purpose, replaceable, and useful on their own. You can add or remove pieces as an app changes, while tightly coupled modules belong together.
6. **Distribute Cohesively.** Remix keeps the ecosystem easy to learn and use by distributing the composable pieces through one `remix` package and one documentation path.

The fastest way to understand Remix is to build the request path once: create a server, define routes, connect a router, and return a response.

## Quickstart: create and run a Remix app {#quickstart}

The quickest way to start a Remix app is with the `remix` CLI. It creates a small full-stack app with a Node server, typed routes, render middleware, a server-rendered home page, source-served assets, and one hydrated component.

```sh
npx remix@next new my-remix-app
```

Install the project dependencies:

```sh
cd my-remix-app
npm i
```

Open `package.json` and you will see one runtime dependency: `remix`. Your app uses that package for the server adapter, router, middleware, UI, assets, and client entry.

Start the dev server:

```sh
npm run dev
```

The server prints the local URL when it starts. By default, the dev server listens on [http://localhost:44100](http://localhost:44100).

![Default Remix home page](/images/app/actions/docs/chapters/01-start-here/remix-template.png)

Your new app includes an Agent Skill and a home page with prompts you can copy into an agentic coding environment. Try one if you want to explore on your own. When you are ready, come back here and we'll take a tour of the files.

## Project tour: server.ts, app/routes.ts, app/router.ts, actions, UI, assets {#project-tour}

A typical Remix application is organized around the request flow. An HTTP request enters `server.ts`, is handed to `app/router.ts`, runs through middleware, is matched against the route definitions in `app/routes.ts`, is handled by a controller action in `app/actions/`, and returns a Web `Response`. Often, that response is HTML generated by a Remix component, but the same path can return redirects, JSON, files, streams, errors, or any other Web response.

That flow gives all of your code a clear place to live. Keep route definitions and URL helpers in routes, connect middleware/routes/controllers in the router, run cross-cutting request logic in middleware, put route-specific behavior in actions, share views and layouts through UI, and put browser behavior in assets.

```txt
my-remix-app/
├── server.ts         # request entrypoint for this runtime
└── app/
    ├── routes.ts     # route definitions, route map, and URL helpers
    ├── router.ts     # middleware, routes, and controller mapping
    ├── middleware/   # request middleware and context providers
    ├── actions/      # controllers and route actions
    ├── ui/           # shared views, layouts, and Remix components
    ├── assets.ts     # browser module server configuration
    └── assets/       # browser entrypoints and client behavior
```

### `server.ts` adapts HTTP to `Request` and `Response`

The server file is the host adapter. Node receives the HTTP request, `createRequestListener` turns it into a Web `Request`, and Remix returns a Web `Response` through `router.fetch(request)`.

```ts filename=server.ts
http.createServer(createRequestListener((request) => router.fetch(request))).listen(44100)
```

Keep this file thin. It is the right place for host-level concerns such as the port, process shutdown, and last-chance error logging, not route behavior.

### `app/routes.ts` defines the URL contract

Routes are ordinary TypeScript values. Your app starts with one page route and one asset route:

```ts filename=app/routes.ts
import { get, route } from 'remix/routes'

export const routes = route({
  assets: get('/assets/*path'),
  home: '/',
})
```

The route object is also how you generate hrefs. As the app grows, use route helpers instead of hand-writing the same URL string in links, redirects, tests, and frame sources.

### `app/router.ts` installs middleware and maps routes to controllers

The router is where the request pipeline comes together. Your app serves files from `public/`, installs render middleware, and maps the route contract to the top-level controller.

```ts filename=app/router.ts
export const router = createRouter<AppContext>({
  middleware: [staticFiles('./public', { index: false }), render()],
})

router.map(routes, controller)
```

Middleware can add request-scoped values to the context before a controller runs. The render middleware adds `context.render`, which lets a controller return server-rendered UI as a normal `Response`.

### `app/actions/` owns route behavior

`app/actions/` is route-owned server code. Despite the name, it is not only for form submissions; it is where controllers, route actions, and route-local helpers live.

```tsx filename=app/actions/controller.tsx
export default createController(routes, {
  actions: {
    home(context) {
      return context.render(<HomePage />)
    },
  },
})
```

Start with the narrowest owner. Put top-level route behavior in `app/actions/controller.tsx`, and add folders like `app/actions/projects/` when a nested route map needs its own controller, UI, middleware, or helpers.

### `app/ui/` holds shared server-rendered UI

`app/ui/` is for UI shared across routes, such as the document shell, app layout, and reusable page pieces. Your app starts with the shared document in `app/ui/document.tsx` and the temporary scaffold page in `app/ui/scaffold-home-page.tsx`.

Route-specific UI can stay near its controller first. Move it to `app/ui/` when more than one route uses it.

### `app/assets.ts` and `app/assets/` serve browser code

`app/assets.ts` configures the source asset server. It serves browser modules from your source tree under `/assets` so client entries can load without adding a bundler step to the development path.

`app/assets/` contains browser-loaded modules. Your app starts with two:

- `app/assets/entry.ts` boots Remix UI in the browser.
- `app/assets/prompt-button.tsx` hydrates the copy-to-clipboard prompt buttons on the home page.

Use `public/` for files that should be served as-is, such as favicons, images, and plain CSS. Use `app/assets/` for TypeScript or JSX modules that Remix should load in the browser.
